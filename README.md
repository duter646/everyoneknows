# EveryoneKnows 世界知识问答网站计划

## 1. 项目目标

打造一个“世界知识”问答网站，满足以下硬性要求：

- 覆盖至少 30 个常识领域。
- 题库总量至少 600 题。
- 题型为选择题，单选为主、多选为辅。
- 每次测试固定 20 题。
- 测试结束可计分。
- 支持上传排行榜。

本期目标采用更高基线：

- 领域数：32 个。
- 题库量：640 题（每领域 20 题）。

---

## 2. 功能范围（MVP）

### 2.1 用户端

- 首页：开始测试、查看规则、进入排行榜。
- 测试页：
	- 每次随机抽取 20 题。
	- 单选题为主（15 题），多选题为辅（5 题）。
	- 显示题号进度、剩余时间（可选）、提交确认。
- 结果页：
	- 展示总分、正确题数、错误题数。
	- 按领域显示正确率（便于用户了解知识短板）。
	- 支持输入昵称并上传成绩到排行榜。

### 2.2 排行榜

- 按分数降序展示 Top N（如 Top 100）。
- 分数相同按提交时间升序（先达到者在前）。
- 展示字段：排名、昵称、分数、正确数、耗时、提交时间。
- 提供今日榜/总榜（MVP 可先做总榜）。

### 2.3 管理端（轻量）

- 题库导入（JSON/CSV）。
- 题目校验（选项完整、答案合法、领域标签合法）。
- 题目启用/停用。

### 3.3 单题数据结构（建议）

```json
{
	"id": "geo_0001",
	"domain": "地理",
	"type": "single",
	"question": "世界上面积最大的国家是？",
	"options": ["加拿大", "俄罗斯", "中国", "美国"],
	"answer": [1],
	"difficulty": "easy",
	"explanation": "按国土面积，俄罗斯位列第一。",
	"tags": ["国家", "面积"]
}
```

说明：

- `type` 取值：`single` 或 `multiple`。
- `answer` 统一用数组表示，单选长度为 1，多选长度大于 1。

## 4. 组卷与出题策略

### 4.1 每次测试 20 题规则

- 15 题单选 + 5 题多选。
- 至少覆盖 10 个不同领域，避免题目过度集中。
- 同一领域单次最多 3 题。
- 难度分布建议：易 3 / 中 4 / 难 3。

### 4.2 抽题算法（MVP）

1. 先按题型和难度构建候选池。
2. 按领域均衡策略抽样（优先抽未出现领域）。
3. 不足时从全局候选池补齐。
4. 对最终 20 题做随机打乱。

---

## 5. 计分与判分方案

### 5.1 基础分

- 单选题：每题 5 分。
- 多选题：每题 5 分。
- 总分满分 100 分。

### 5.2 判分规则

- 单选：选中唯一正确选项得满分，否则 0 分。
- 多选：
	- 必须“全对且不多选”才得满分。
	- 漏选得2分

可选增强（后续迭代）：多选按部分得分规则计分，但排行榜需标注计分版本。

### 5.3 排名规则

排序优先级：

1. 总分降序
2. 正确题数降序
3. 用时升序
4. 提交时间升序

---

## 6. 排行榜上传与风控

### 6.1 上传流程

1. 前端提交：昵称、分数、正确数、用时、答题明细摘要、试卷ID。
2. 后端复算：基于试卷ID和用户答案重新判分。（可以先不用，后端暂时直接用supabase即可）
3. 写入排行榜：仅接受后端复算后的可信成绩。

### 6.2 基础防作弊（先不做，后续有需要添加）

- 前后端分离判分，以后端判分为准。
- 试卷ID签名（防篡改）。
- 单 IP/设备短时提交频率限制。
- 异常成绩标记（极短耗时高分、重复模式）。

## 7. 技术方案（建议）

### 7.1 架构

- 前端：React + Vite + TypeScript
- 后端：Node.js + Express + TypeScript
- 数据库：PostgreSQL（或 MVP 用 SQLite）（以上先用supabase）
- 部署：Vercel（前端）+ Render/Fly.io（后端）

### 7.2 核心表设计（简化）

- `questions`
	- `id`, `domain`, `type`, `question`, `options_json`, `answer_json`, `difficulty`, `status`
- `quiz_sessions`
	- `id`, `paper_seed`, `started_at`, `finished_at`, `duration_sec`
- `quiz_answers`
	- `id`, `session_id`, `question_id`, `selected_json`, `is_correct`, `score`
- `leaderboard`
	- `id`, `nickname`, `score`, `correct_count`, `duration_sec`, `submitted_at`, `session_id`

## 8. 里程碑计划

### M1：需求冻结与数据标准（1-2 天）

- 确认 32 个领域清单。
- 确认题目 JSON Schema。
- 确认评分和排行榜规则。

### M2：题库建设（5-7 天）

- 先完成 320 题（每领域 10 题）可运行版本。
- 完成全量 640 题。
- 完成题库质量抽检（准确性、歧义、重复）。

### M3：核心功能开发（4-6 天）

- 测试流程：抽题、答题、提交、结果页。
- 排行榜上传与展示。
- 后端复算与基础风控。

### M4：测试与上线（2-3 天）

- 功能测试、边界测试、性能测试。
- 上线部署与监控。

总工期（MVP）：约 2-3 周。

---

## 9. 验收标准

- 领域数 >= 30（目标 32）。
- 题库量 >= 600（目标 640）。
- 每次测试固定 20 题。
- 支持单选与多选，单选占比明显更高。
- 能稳定计分并展示结果。
- 支持上传排行榜，且后端可复算成绩。
- 主要页面在桌面端和移动端可用。

---

## 10. 风险与应对

- 风险：题目事实错误或争议。
	- 应对：设置审核流程、来源标注、争议题快速下线。
- 风险：题目重复导致体验下降。
	- 应对：题干相似度检查 + 领域均衡抽题。
- 风险：排行榜被脚本刷榜。
	- 应对：后端复算、限流、异常检测、必要时验证码。

---
